    Issue 1: Deliver FastMoq analyzers through the aggregate package and a standalone analyzer package

    Suggested milestone: MVP

    Suggested labels: enhancement, analyzers, packaging, v4.x

    Problem:

    FastMoq guidance is most useful when it appears automatically in the common consumer path, but the package boundaries still matter.

    If analyzers ship only as a separate package, many aggregate-package consumers will never know to add them. If analyzers ship in core, the lighter package stops being light and every direct core consumer inherits editor and build policy by default.

    Proposed change:

    Publish FastMoq.Analyzers as a standalone package, keep FastMoq.Core analyzer-free, and also include the analyzer assets in the aggregate FastMoq package.

    That gives aggregate-package consumers zero-config guidance while still preserving explicit control for direct core consumers.

    Document the opt-out path for aggregate-package consumers through ExcludeAssets="analyzers".

    Acceptance criteria:

    • Referencing FastMoq loads the analyzer assets automatically.
    • Referencing FastMoq.Core does not load analyzer assets automatically.
    • FastMoq.Analyzers can still be referenced directly by FastMoq.Core consumers.
    • Documentation explains the difference between FastMoq, FastMoq.Core, and FastMoq.Analyzers.
    • Aggregate-package consumers can disable analyzer loading with ExcludeAssets="analyzers".
    • Pack validation confirms the analyzer assets are present in both the standalone analyzer package and the aggregate package.

    Notes:

    • Do not rely on a transitive package dependency from FastMoq to FastMoq.Analyzers. Analyzer assets are private by default in normal PackageReference flows, so the aggregate package should pack the analyzer assets directly.
    • This issue is about delivery and discoverability, not about expanding the analyzer rule set by itself.

    Issue 2: Ship high-confidence migration analyzers and code fixes for common provider-first rewrites

    Suggested milestone: MVP

    Suggested labels: enhancement, migration, analyzers, v4.x

    Problem:

    Real migrations repeatedly stall on a small set of mechanical edits that are cheap to describe but expensive to apply across shared helpers and large suites.

    One outdated helper boundary can force repetitive leaf-test churn.

    Proposed change:

    Ship the first analyzer wave as a high-confidence migration pass with deterministic code fixes.

    Initial diagnostics should cover:

    • tracked-mock .Object usage and suggest .Instance or GetObject<T>()
    • raw provider-native Reset() calls when IFastMock.Reset() is the provider-first replacement
    • VerifyLogger(...) usage when VerifyLogged(...) is the provider-safe replacement
    • helper layers or files that mix GetMock<T>() with GetOrCreateMock<T>() and can be normalized mechanically

    Acceptance criteria:

    • The analyzer package can be consumed from ordinary test projects.
    • Each initial diagnostic ships with at least one code fix.
    • Diagnostics are scoped so intentionally Moq-specific tests are not flooded with warnings by default.
    • The docs include a short section explaining that aggregate-package consumers get the analyzer automatically and core consumers add FastMoq.Analyzers explicitly.
    • Complex Moq-only constructs such as SetupSet(...), SetupAllProperties(), Protected() chains, and out / ref verification remain out of scope for automatic fixes.

    Notes:

    • This issue is intentionally the MVP slice, not the full analyzer backlog.
    • The goal is to remove the highest-volume mechanical churn first.
    • Dedicated SetupSet(...) migration support is tracked separately in Issue 11 rather than in the initial automatic-fix wave.

    Issue 3: Expand migration analyzers for helper-boundary and compatibility cleanup

    Suggested milestone: Follow-up

    Suggested labels: enhancement, migration, analyzers, v4.x

    Problem:

    After the high-confidence migration pass, the remaining migration churn still clusters around helper boundaries and compatibility aliases rather than leaf-test edits.

    The migration docs already identify these patterns, but the analyzer MVP would not yet cover them.

    Proposed change:

    Add the second migration-analyzer wave for compatibility-surface cleanup and helper-boundary modernization.

    Follow-up diagnostics should cover:

    • helper signatures built around raw Times or Func<Times> and suggest TimesSpec
    • Initialize<T>() as a migration target rather than a preferred setup entry point
    • MockOptional as a compatibility alias that should move to explicit optional-parameter controls
    • Strict compatibility guidance when the code should move toward clearer preset or policy-based intent
    • remaining legacy GetMock<T>() usage in provider-first helper layers where the replacement is clearer than preserving the old boundary

    Acceptance criteria:

    • Each follow-up diagnostic is backed by analyzer tests that model helper-layer usage, not only leaf tests.
    • Deterministic fixes are provided where symbol replacement is safe.
    • Ambiguous guidance ships as suggestion-level or info-level rather than an aggressive warning.
    • The migration docs and analyzer wording stay aligned on what is guidance versus what is a compatibility-only escape hatch.

    Notes:

    • SetupSet(...), SetupAllProperties(), arbitrary Protected() interception, and out / ref verification are still acceptable migration exceptions and should remain out of automatic-fix scope.
    • If FastMoq adds a first-party path for the common setter-observation scenarios, track that separately from analyzer auto-fixes so the migration guidance stays explicit about what still needs raw Moq. See Issue 11.

    Issue 4: Add provider-selection diagnostics and bootstrap guidance for provider-specific APIs

    Suggested milestone: Follow-up

    Suggested labels: enhancement, migration, providers, analyzers, v4.x

    Problem:

    Real migrators often install a provider package and import its namespace, but still forget that reflection remains the active default until registration changes it.

    The result is confusing failures when code reaches GetMock<T>(), VerifyLogger(...), AsMoq(), AsNSubstitute(), Protected(), or other provider-specific APIs under the wrong active provider.

    Proposed change:

    Standardize provider-mismatch diagnostics across provider-specific entry points and add analyzer guidance for missing provider bootstrap.

    The work should cover both runtime and analyzer paths:

    • runtime diagnostics should clearly name the active provider, the expected provider, and the bootstrap action
    • analyzer guidance should detect provider-specific API usage in projects that do not select the required provider and offer framework-specific bootstrap guidance where feasible

    Acceptance criteria:

    • Provider-specific APIs fail with a consistent, actionable runtime message when the required provider is not active.
    • The runtime message names both the active provider and the expected provider.
    • The guidance clearly separates package installation, namespace import, provider registration, and default-provider selection.
    • At least one analyzer test covers the missing-bootstrap path for Moq and one optional provider.
    • At least one integration test covers the standardized runtime message path for Moq and one optional provider.

    Notes:

    • The goal is better failure guidance, not a change to the default-provider contract.
    • If the analyzer code fix is too brittle to automate fully, it should still provide a clear framework-specific snippet or direct link target.

    Issue 5: Add authoring-guidance analyzers for new FastMoq tests

    Suggested milestone: Follow-up

    Suggested labels: enhancement, analyzers, usability, testing, v4.x

    Status in current branch:

    • Initial rule wave implemented and validated: FMOQ0010 for typed provider extensions over raw native access, FMOQ0011 for FastMoq.Web helper discovery, and FMOQ0012 for provider-neutral HTTP helper guidance.
    • Remaining guidance items in this issue still need design and implementation so the analyzer family stays low-noise and intentionally scoped.

    Problem:

    The current docs contain strong guidance for how new FastMoq tests should be written, but that guidance lives mostly in markdown and API docs rather than in the editor.

    As a result, new tests still drift toward older or hand-rolled patterns even when a better first-party shape already exists.

    Proposed change:

    Add a second analyzer family focused on low-noise authoring guidance for new tests.

    Initial suggestion-level or info-level diagnostics should cover the most stable documented rules:

    • prefer GetOrCreateMock<T>() over AddType(...) when the dependency is still conceptually a mock
    • prefer AddType(...) or AddKnownType(...) when resolution is being replaced intentionally
    • prefer VerifyLogged(...) for new logger assertions
    • prefer typed provider extensions over raw NativeMock or GetNativeMock(...) when possible
    • prefer FastMoq.Web helpers over hand-rolled HttpContext, ControllerContext, and IHttpContextAccessor setup
    • prefer a typed IServiceProvider helper over one-object-for-all-types service-provider shims
    • prefer WhenHttpRequest(...) or WhenHttpRequestJson(...) over protected HttpMessageHandler interception when the test only cares about request and response behavior

    Acceptance criteria:

    • Authoring-guidance diagnostics default to suggestion or info severity.
    • Guidance diagnostics are easy to suppress when a test is intentionally choosing a different shape.
    • The analyzer package does not flood existing suites with stylistic noise by default.
    • Docs explain the intended role of the guidance rules and how to tune their severity.

    Notes:

    • This issue is about guidance, not general code-style enforcement.
    • The analyzer should prefer teaching a documented FastMoq pattern over inventing new policy.

    Issue 6: Add a typed IServiceProvider test helper for framework-heavy suites

    Suggested milestone: Follow-up

    Suggested labels: enhancement, testing, infrastructure, v4.x

    Problem:

    Framework-heavy test suites often keep ad hoc IServiceProvider shims that return the same object for every GetService(Type) request.

    Those shims are easy to write, but they hide type-resolution bugs and can mask cast failures until much later in execution.

    Proposed change:

    Add a FastMoq helper that builds a type-aware IServiceProvider for tests.

    The helper should support:

    • building from an existing IServiceCollection
    • lightweight registration of concrete instances, factories, and simple test doubles
    • common framework primitives such as logging and options without requiring callers to hand-roll GetService(Type) switches
    • preserving keyed registrations when the backing container supports them

    Acceptance criteria:

    • The helper resolves services by requested type rather than returning a single placeholder object.
    • Tests can supply instances, factories, or a prebuilt IServiceCollection.
    • Keyed registrations are preserved when the underlying container supports them.
    • The helper has executable examples in the repo.
    • The final API makes the package boundary explicit instead of accidentally widening unrelated core dependencies.

    Notes:

    • This helper should stay provider-neutral.
    • The goal is test harness setup, not a replacement for the main application DI container.

    Issue 7: Add an Azure Functions worker helper for FunctionContext.InstanceServices

    Suggested milestone: Follow-up

    Suggested labels: enhancement, azure-functions, testing, v4.x

    Problem:

    Azure Functions worker migrations hit a specific trap around FunctionContext.InstanceServices. A naive shim that returns one object for every service request can keep old tests limping along while hiding failures for typed services such as IOptions<WorkerOptions>.

    Proposed change:

    Add a first-party helper for wiring FunctionContext.InstanceServices from a type-aware service provider.

    The helper should:

    • build on the typed IServiceProvider helper if Issue 6 lands first
    • allow tests to attach a real or type-aware service provider to FunctionContext
    • include examples that cover ILoggerFactory, options resolution, and ordinary typed service retrieval
    • keep the Azure Functions dependency in an intentional package boundary rather than pushing it into unrelated core paths

    Acceptance criteria:

    • The helper does not rely on a one-object-for-all-types GetService(Type) shim.
    • The repo includes executable coverage for typed options and logging resolution through InstanceServices.
    • The final API makes the package dependency story explicit.
    • Migration guidance can point to one recommended helper instead of ad hoc shims.

    Notes:

    • This should stay optional and framework-specific.
    • The goal is to remove repeated fragile harness code from consumer suites.

    Issue 8: Improve provider-first tracked-mock discoverability in IntelliSense and API docs

    Suggested milestone: MVP

    Suggested labels: enhancement, usability, api-docs, v4.x

    Status in current branch:

    • Implemented and validated.

    Problem:

    Recent migrators did not discover that IFastMock<T> already exposes Instance and that tracked mocks already expose Reset().

    That suggests the provider-first path is present but not obvious enough during everyday use.

    Proposed change:

    Tighten XML documentation and generated API examples for the provider-first tracked-mock surface.

    Focus areas:

    • IFastMock
    • IFastMock<T>
    • GetOrCreateMock<T>()
    • GetObject<T>()
    • NativeMock
    • provider adapter entry points such as AsMoq() and AsNSubstitute()

    The documentation should call out the common migration replacements for .Object, raw Reset(), and direct provider-native access.

    Acceptance criteria:

    • IntelliSense for GetOrCreateMock<T>() and IFastMock<T>.Instance explicitly mentions the common .Object replacement path.
    • IntelliSense for Reset() explicitly describes the provider-first reset path.
    • Generated API docs include at least one concise provider-first example and one provider-native escape-hatch example.
    • The examples do not blur the provider-neutral versus provider-specific boundary.

    Notes:

    • This issue is about discoverability, not adding a new Moq-shaped alias surface.
    • The goal is to help both migrators and new test authors find the provider-first path sooner.

    Issue 9: Add keyed-service diagnostics for same-type constructor dependencies

    Suggested milestone: Follow-up

    Suggested labels: enhancement, migration, analyzers, keyed-services, v4.x

    Problem:

    Recent migration feedback exposed a subtle false-positive success case: the production constructor consumed the same interface twice, differentiated by DI service keys, but the tests used one unkeyed type-based mock.

    Those tests still passed, but they could not catch a public/private or primary/secondary swap because the two production dependencies had been collapsed into one double.

    The keyed FastMoq support already exists in v4 through AddKeyedType(...), GetKeyedObject<T>(), and GetOrCreateMock<T>(MockRequestOptions) with MockRequestOptions.ServiceKey. The missing piece is early diagnostics.

    Proposed change:

    Add a targeted diagnostic for same-type constructor dependencies that are disambiguated by keyed DI attributes.

    The diagnostic path should:

    • recognize constructor parameters marked with keyed DI attributes supported by FastMoq resolution, such as FromKeyedServices
    • warn when a FastMoq test targets that constructor through MockerTestBase<T> or CreateInstance<T>() but only sets up the duplicated service type through unkeyed GetMock<T>(), unkeyed GetOrCreateMock<T>(), or unkeyed AddType<T>()
    • explain that one type-based double collapses multiple production dependencies into one test double
    • suggest keyed FastMoq alternatives such as GetOrCreateMock<T>(new MockRequestOptions { ServiceKey = ... }), AddKeyedType(...), GetKeyedObject<T>(), or explicit constructor injection with separate doubles
    • link directly to the keyed migration and testing guidance

    Acceptance criteria:

    • The diagnostic can identify duplicate constructor service types that are disambiguated by keyed DI attributes.
    • The warning message explains the specific risk of collapsing multiple production dependencies into one test double.
    • The message suggests at least one keyed FastMoq pattern and one explicit-constructor-injection alternative.
    • Tests that already use keyed options or keyed registrations for the duplicated service type do not trigger the warning.
    • The docs include a short keyed-services section that the diagnostic can link to.

    Notes:

    • Static analysis can catch the structural risk in the constructor and the test arrangement, but it cannot prove every business-level selection semantic automatically.
    • The goal is to surface the ambiguity early, not to replace a small wiring or integration test when the DI contract itself also needs coverage.

    Issue 10: Analyze and stage a bUnit upgrade for FastMoq's Blazor test helpers

    Suggested milestone: Follow-up

    Suggested labels: enhancement, dependencies, blazor, bunit, testing, v4.x

    Problem:

    FastMoq.Web currently references bunit 1.38.5, while the current stable NuGet line has moved to 2.7.2.

    FastMoq's Blazor layer uses bUnit directly through Bunit, Bunit.Rendering, Bunit.TestDoubles, and Bunit.Web.AngleSharp, with wrapper and helper surfaces in:

    • FastMoq.Web/Blazor/MockerBlazorTestBase.cs
    • FastMoq.Web/Blazor/Models/ComponentState.cs
    • FastMoq.Web/Blazor/Interfaces/IMockerBlazorTestHelpers.cs
    • FastMoq.Tests.Web

    A direct package bump could break the current Blazor helper surface, so this needs an analysis-first pass instead of an unscoped dependency update.

    Proposed change:

    Analyze the upgrade path from the current bUnit dependency to the latest supported stable line, record the API deltas that matter to FastMoq, and determine whether the upgrade can ship as a direct version bump or needs a staged compatibility pass.

    The analysis should cover:

    • package-version delta and any relevant runtime or framework alignment changes
    • compile-time and behavioral impact on MockerBlazorTestBase, ComponentState, and helper interfaces
    • authorization, navigation, AngleSharp, and JS-interop touchpoints currently used by FastMoq's wrappers and tests
    • whether FastMoq's public Blazor helper APIs need compatibility shims or signature changes before upgrading

    Acceptance criteria:

    • The current bUnit usage surface in FastMoq is inventoried and documented.
    • The candidate upgrade target and its expected framework/runtime constraints are recorded.
    • A compile-and-test spike is completed against the selected bUnit version, or concrete blockers are documented with exact API breaks.
    • Any required wrapper or helper updates are identified before the package version is changed.
    • If the upgrade is approved, the eventual implementation work updates Directory.Packages.props, affected tests, and any necessary docs in the same change set or in explicitly linked follow-up work.

    Notes:

    • Keep the bUnit dependency story confined to FastMoq.Web and FastMoq.Tests.Web; do not widen it into unrelated packages.
    • This issue is analysis-first. If the surface breaks are broad, the actual version upgrade can be split into a dedicated implementation item.

    Issue 11: Add a first-party migration path for SetupSet(...)-heavy tests

    Suggested milestone: Follow-up

    Suggested labels: enhancement, migration, testing, mocks, v4.x

    Problem:

    SetupSet(...) remains one of the most visible Moq-only blockers in real v4 migrations.

    Today the docs can only say "keep that test on Moq" or "replace it with a fake or stub," which is accurate but leaves a recurring migration gap untracked in the roadmap.

    The gap shows up most often in property-setter and cache-entry tests, especially around patterns such as IMemoryCache, ICacheEntry, or other collaborators where the important behavior is "capture the assigned value" rather than "stay on Moq forever."

    Proposed change:

    Design and add a first-party migration path for the common SetupSet(...) scenarios without trying to emulate every Moq setter-interception edge case.

    The work can land as one focused helper or a small helper family, but it should target the migration cases that are currently frequent and repetitive:

    • observing or capturing assigned property values in a test-friendly way
    • replacing simple SetupSet(...) assertions with a recommended fake, helper, or provider-neutral pattern when the test only needs setter observation
    • documenting which setter-heavy patterns should still remain on raw Moq because the semantics are broader than simple value capture

    Acceptance criteria:

    • The repo defines one recommended first-party migration path for the common SetupSet(...) scenarios instead of leaving only ad hoc guidance.
    • At least one executable example covers a straightforward setter-observation case and one cache-entry or similar framework-heavy case.
    • The docs clearly distinguish between the new recommended path and the cases that should still remain on raw Moq.
    • The final API does not try to recreate the full Moq setter-interception surface in provider-neutral core if a narrower helper is clearer.
    • Migration guidance, provider-capability docs, and analyzer wording remain aligned on when SetupSet(...) is still an intentional exception.

    Notes:

    • This issue is about closing the most common migration gap, not about cloning all of Moq's property APIs.
    • If the cleanest solution is a focused fake or helper pattern rather than a new mock-provider abstraction, prefer the simpler shape.
